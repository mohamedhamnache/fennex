Overview & vision

01 One platform for the entire content lifecycle

Most teams stitch together five tools — a keyword researcher, a writer, an image generator, a publisher and a rank tracker — and copy-paste between them. Fennex collapses that whole workflow into a single, coherent product where every step is aware of the others.

The problem it solves

Producing content that actually ranks is slow and fragmented. You research keywords in one tab, draft in another, hand-optimize for SEO by memory, generate images somewhere else, publish manually, and only weeks later discover whether any of it worked. Generic AI writers make it faster to produce mediocre content — but mediocre content doesn't rank.

Fennex is built on a simple conviction: AI should do the work, grounded in your real data, and the whole pipeline should live in one place. Every article is scored against a real on-page rubric as you write, grounded in your actual Search Console queries and tracked keywords, and one click away from being published and monitored.

What you can do — the three pillars

Create

Generate long-form SEO articles that stream in live, social posts, on-brand images, product shots and marketing banners. Edit everything in a rich studio with an agentic co-writer beside you.

Optimize

Research and cluster keywords by real demand, score every page on-page in real time, run technical site audits, scan competitors and track your live SERP positions daily.

Publish & grow

Push straight to WordPress and social, plan on a calendar, then watch rankings and get alerts when positions move — closing the loop from idea to measured result.

How a request flows

The browser calls the FastAPI API with a JWT. Quick reads return immediately. Anything slow — generating an article, crawling a competitor, tracking ranks — is either streamed back token-by-token over Server-Sent Events, or enqueued to the arq worker via Redis and polled/pushed when done. The worker and API share the same models and services, so business logic lives in one place.

Key architectural decisions

  • Async everywhere — FastAPI + SQLAlchemy 2 async + asyncpg; long tasks run on arq workers so requests stay fast and the UI stays responsive.
  • Real-time streaming — article generation and the Dune co-writer stream over SSE; the frontend renders tokens as they arrive.
  • Deterministic + AI split — SEO scoring, on-page checks and internal-link discovery are deterministic (the exact formula is mirrored in TypeScript for instant client feedback); generation and rewriting are AI, grounded in real data.
  • Multi-tenant & secure — JWT with refresh tokens, every query org-scoped, third-party credentials encrypted at rest with ENCRYPTION_KEY.
  • Provider-agnostic AI — a router picks the best available model per task (Anthropic → OpenAI → Google) with a quality-aware fallback, so long-form never silently drops to a weak model.

Get running

07 Local development

The full stack runs in Docker in a couple of commands.

Prerequisites

  • Docker & Docker Compose
  • Node 18+ and pnpm (to run the frontend outside Docker)
  • Python 3.11+ (optional, for API tooling)

  1. Clone & configure

    Copy the example env and set at least the secret keys. AI provider keys are added later in the app's Settings.

    bash
    git clone <repo> fennex && cd fennex
cp .env.example .env
# edit .env — set SECRET_KEY and ENCRYPTION_KEY

  2. Start everything (Docker)

    Brings up Postgres, Redis, the API, the arq worker, the web app and the crawler.

    bash
    docker compose up --build
# or the Makefile shortcut
make dev

  3. Run database migrations

    bash
    make db-migrate      # apply Alembic migrations in the api container
# make db-reset      # drop, recreate and re-migrate

  4. Open the apps

    Frontend → http://localhost:3000 · API & interactive docs → http://localhost:8000/docs · Crawler → http://localhost:8001

  5. Verify changes

    The frontend has no test runner — verify with a typecheck. The API is covered by pytest.

    bash
    docker compose exec api pytest -q
docker compose exec web npx tsc --noEmit

08 Deploy · Vercel + Supabase

The fastest path to production. Vercel hosts the frontend, Supabase provides Postgres, and a container host runs the Python API + worker.

Honest architecture:Vercel is ideal for the Next.js frontend and Supabase is a managed Postgres (with pgvector). Fennex's backend is a long-running FastAPI service plus an arq worker and Redis — those don't fit Vercel's serverless model, so run the API on a container host (Render, Railway or Fly.io) and use Upstash for Redis. This split is production-grade and inexpensive.

Step by step

  1. Create the Supabase database

    New Supabase project → enable the vector extension and copy the connection string (use the connection pooler URL for serverless-friendly access).

    SQL · Supabase SQL editor
    create extension if not exists vector;
    DATABASE_URL (asyncpg)
    postgresql+asyncpg://postgres:<pwd>@db.<ref>.supabase.co:5432/postgres

  2. Provision Redis (Upstash)

    Create an Upstash Redis database and copy its rediss:// URL into REDIS_URL.

  3. Deploy the API + worker (Render example)

    Point Render at the repo and build from apps/api/Dockerfile. Create two services from the same image: a Web Service (uvicorn) and a Background Worker (arq).

    API service · start command
    uvicorn app.main:app --host 0.0.0.0 --port $PORT
    Worker service · start command
    arq app.workers.worker.WorkerSettings
    run migrations once (release / shell)
    alembic upgrade head

  4. Set backend env vars

    On both services set DATABASE_URL, REDIS_URL, SECRET_KEY, ENCRYPTION_KEY, S3 credentials and Stripe keys — see the Environment reference.

  5. Deploy the frontend to Vercel

    Import the repo, set the root to apps/web, framework Next.js, install command pnpm install. Add:

    Vercel · Environment Variables
    NEXT_PUBLIC_API_URL = https://your-api.onrender.com
NEXTAUTH_URL        = https://your-app.vercel.app
    Monorepo:In Vercel project settings set the Root Directory to apps/web so the workspace packages resolve.

  6. Wire CORS & go live

    Add the Vercel domain to the API's allowed origins, then create an org, add an AI key in Settings and generate your first article.

Step by step

  1. Provision managed data

    Create an RDS PostgreSQL instance (enable vector), an ElastiCache Redis cluster and an S3 bucket, in a VPC with private subnets.

    psql · once
    CREATE EXTENSION IF NOT EXISTS vector;

  2. Build & push images to ECR

    bash
    aws ecr create-repository --repository-name fennex-api
aws ecr get-login-password | docker login --username AWS --password-stdin <acct>.dkr.ecr.<region>.amazonaws.com

docker build -t fennex-api apps/api
docker tag fennex-api:latest <acct>.dkr.ecr.<region>.amazonaws.com/fennex-api:latest
docker push <acct>.dkr.ecr.<region>.amazonaws.com/fennex-api:latest

  3. Store secrets

    Create a Secrets Manager secret with DATABASE_URL, REDIS_URL, SECRET_KEY, ENCRYPTION_KEY, Stripe & S3 values; reference it from ECS task definitions.

  4. Create the ECS cluster & services

    One Fargate cluster with three services from the API image: api (behind an ALB on 8000), worker (arq …) and crawler. The ALB terminates TLS with an ACM certificate.

    task command · worker container
    arq app.workers.worker.WorkerSettings

  5. Run migrations

    Run a one-off ECS task (or exec into the API task) with alembic upgrade head.

  6. Deploy the frontend

    Connect the repo to Amplify Hosting (root apps/web) or build next build and serve via S3 + CloudFront. Set NEXT_PUBLIC_API_URL to the HTTPS ALB domain and add that domain to the API CORS allow-list.

  7. Harden

    Least-privilege security groups (ALB → API → RDS/Redis only), RDS automated backups, CloudWatch logs & alarms, and autoscaling on the ECS services.

Tip:The repo ships docker-compose.prod.yml (postgres, redis, api, worker, web) — a faithful blueprint of the production topology and a great reference when authoring ECS task definitions or a single-VM Docker deployment.
