Fennex Documentation

Fennex is an AI-powered SEO & content platform. It helps teams research keywords, generate and optimize articles, create and edit images, publish to WordPress & social, and track rankings — orchestrated by a pack of seven specialised AI agents.

Next.js 14 · React 18 FastAPI · Python 3.11 PostgreSQL + pgvector Redis · arq workers Anthropic · OpenAI 6 languages

01 Overview

What Fennex is, at a glance.

Fennex is a multi-tenant SaaS: an organization owns one or more projects (each a website), and every capability — content, SEO, images, social, analytics — is scoped to a project. AI work is performed by named agents (the Fennex Pack) that share a common LLM layer and are grounded in the project's real data (Google Search Console, tracked keywords, crawled competitors).

Long-form SEO articles, social posts, images and marketing banners — generated, edited and scored in-app.

Key architectural decisions

  • Async everywhere — FastAPI + SQLAlchemy 2 async + asyncpg; long tasks (article generation, crawling, rank tracking) run on arq workers backed by Redis, so requests stay fast.
  • Real-time streaming — article generation and the Dune co-writer stream tokens to the browser over Server-Sent Events.
  • Deterministic + AI — SEO scoring, on-page checks and internal-link discovery are deterministic (identical formula mirrored in TS for instant client feedback); generation and rewriting are AI, grounded in the project's real data.
  • Multi-tenant & secure — JWT auth with refresh tokens; every query is org-scoped; third-party credentials are encrypted at rest (AES via ENCRYPTION_KEY).
  • Provider-agnostic AI — a router picks the best available model per task (Anthropic → OpenAI → Google) with a quality-aware fallback.

06 Local development

Get the full stack running in minutes.

Prerequisites

  • Docker & Docker Compose
  • Node 18+ and pnpm (for running the frontend outside Docker)
  • Python 3.11+ (optional, for API tooling)

  1. Clone & configure

    Copy the example env and fill in at least a database URL, secret keys and one AI provider key.

    bash
    git clone <repo> fennex && cd fennex
cp .env.example .env
# edit .env — set SECRET_KEY, ENCRYPTION_KEY and an AI key in Settings later

  2. Start everything (Docker)

    Brings up Postgres, Redis, the API, the arq worker, the web app and the crawler.

    bash
    docker compose up --build
# or the Makefile shortcut
make dev

  3. Run database migrations

    bash
    make db-migrate      # apply Alembic migrations inside the api container
# make db-reset      # drop, recreate and re-migrate

  4. Open the apps

    Frontend → http://localhost:3000 · API & docs → http://localhost:8000 (/docs) · Crawler → http://localhost:8001

  5. Verify a change

    The frontend has no test runner — verify with pnpm --filter @fennex/web typecheck. The API is tested with pytest.

    bash
    docker compose exec api pytest -q
docker compose exec web npx tsc --noEmit

07 Deploy · Vercel + Supabase

The fastest path to production. Vercel hosts the frontend, Supabase provides Postgres, and a container host runs the Python API + worker.

ℹ Honest architecture:Vercel is ideal for the Next.js frontend, and Supabase is a managed Postgres (with pgvector). Fennex's backend is a long-running FastAPI service plus an arq worker and Redis — those don't fit Vercel's serverless model, so run the API on a container host (Render, Railway or Fly.io) and use Upstash for Redis. This split is production-grade and cheap.

Step by step

  1. Create the Supabase database

    New Supabase project → Database. Enable the vector extension and grab the connection string (use the connection pooler URL for serverless-friendly access).

    SQL · Supabase SQL editor
    create extension if not exists vector;
    DATABASE_URL (asyncpg)
    postgresql+asyncpg://postgres:<pwd>@db.<ref>.supabase.co:5432/postgres

  2. Provision Redis (Upstash)

    Create an Upstash Redis database and copy its rediss:// URL into REDIS_URL.

  3. Deploy the API + worker (Render example)

    Point Render at the repo, build from apps/api/Dockerfile. Create two services from the same image: a Web Service (uvicorn) and a Background Worker (arq).

    API service · start command
    uvicorn app.main:app --host 0.0.0.0 --port $PORT
    Worker service · start command
    arq app.workers.worker.WorkerSettings
    run migrations once (release / shell)
    alembic upgrade head

  4. Set backend env vars

    On both API + worker services set: DATABASE_URL, REDIS_URL, SECRET_KEY, ENCRYPTION_KEY, S3 credentials, Stripe keys. See the Environment reference.

  5. Deploy the frontend to Vercel

    Import the repo, set the root to apps/web, framework Next.js. Add the env var pointing at your API:

    Vercel · Environment Variables
    NEXT_PUBLIC_API_URL = https://your-api.onrender.com
NEXTAUTH_URL        = https://your-app.vercel.app
    ⚠ Monorepo:In Vercel project settings set the Root Directory to apps/web and the install command to pnpm install so workspace packages resolve.

  6. Wire CORS & go live

    Add the Vercel domain to the API's allowed origins, point the API at Supabase & Upstash, then create an org, add an AI key in Settings and generate your first article.

Step by step

  1. Provision managed data

    Create an RDS PostgreSQL instance (enable vector), an ElastiCache Redis cluster, and an S3 bucket. Put them in a VPC with private subnets.

    psql · once
    CREATE EXTENSION IF NOT EXISTS vector;

  2. Build & push images to ECR

    bash
    aws ecr create-repository --repository-name fennex-api
aws ecr get-login-password | docker login --username AWS --password-stdin <acct>.dkr.ecr.<region>.amazonaws.com

docker build -t fennex-api apps/api
docker tag fennex-api:latest <acct>.dkr.ecr.<region>.amazonaws.com/fennex-api:latest
docker push <acct>.dkr.ecr.<region>.amazonaws.com/fennex-api:latest

  3. Store secrets

    Create a Secrets Manager secret holding DATABASE_URL, REDIS_URL, SECRET_KEY, ENCRYPTION_KEY, Stripe & S3 values; reference it from ECS task definitions.

  4. Create the ECS cluster & services

    One Fargate cluster with three services from the API image: api (behind an ALB, port 8000), worker (arq …), and crawler. The ALB terminates TLS (ACM cert) and routes to the API target group.

    task command · worker container
    arq app.workers.worker.WorkerSettings

  5. Run migrations

    Run a one-off ECS task (or exec into the API task) with alembic upgrade head.

  6. Deploy the frontend

    Connect the repo to Amplify Hosting (root apps/web) or build next build and serve via S3 + CloudFront. Set NEXT_PUBLIC_API_URL to the ALB/HTTPS domain and add that domain to the API CORS allow-list.

  7. Harden

    Security groups least-privilege (ALB → API → RDS/Redis only), enable RDS automated backups, CloudWatch logs & alarms, and autoscaling policies on the ECS services.

✓ Tip:The repo ships docker-compose.prod.yml (postgres, redis, api, worker, web). It's a faithful blueprint of the production topology — a great reference when writing ECS task definitions or a single-VM Docker deployment.
